# Few-shot 与 Chain-of-Thought

> 核心问题：如何让 LLM 输出更准确、更符合预期？答案是 **给例子 (Few-shot)** 和 **展示推理过程 (CoT)**。

## 1. 从 Zero-shot 到 Few-shot

### Zero-shot：不给例子，直接问

```
用户：这篇文章的 sentiment 是什么？"等了 3 小时才上菜。"
AI：负面
```

模型依靠预训练阶段学到的"sentiment"概念来理解任务。简单任务还行，复杂任务容易翻车。

### Few-shot：给几个例子，让模型"看清格式再答题"

```
用户：判断 sentiment。positive/negative/neutral。

示例 1：服务太棒了 → positive
示例 2：等了半小时才上菜 → negative
示例 3：还行吧，无功无过 → neutral

现在判断：装修很好看但菜一般般 → ?
AI：neutral
```

**Few-shot 的核心机制**：不是"教会"模型，而是**激活模型已有的能力并校准输出格式**。模型在预训练阶段已经学会了 sentiment 分析，few-shot 只是告诉它"用这种格式做这道题"。

### Few-shot 的实践边界

| 能力类型 | Zero-shot 够用？ | 需要 Few-shot？ |
|---|---|---|
| 情感分析、摘要 | ✅ 够用 | 格式特殊时才需要 |
| 分类（已知类别） | ✅ 够用 | 类别名不直观时需要 |
| 结构化输出（JSON/XML） | ⚠️ 简单结构还行 | ✅ 复杂 schema 必须给例子 |
| 代码生成（特定风格） | ❌ 风格随机 | ✅ 必须给代码示例 |
| 数学推理 | ❌ 容易错 | ✅ 需要 CoT 示例 |

### 什么时候 Few-shot 不灵？

1. **例子太相似** → 模型过度模仿，失去泛化
2. **例子有矛盾** → 模型困惑，输出不稳定
3. **例子太多（>10）** → 收益递减，浪费 context window
4. **任务本身超出模型能力** → 给再多例子也没用

## 2. Chain-of-Thought (CoT) 推理

### CoT 是什么？

让模型**先推理再回答**，而不是直接蹦出一个答案。

**不用 CoT（标准 prompting）：**
```
Q: 小明有 5 本书，他买了 3 本，又送了 2 本给朋友，还剩几本？
A: 6 本
```
模型可能猜对，也可能猜错——因为它没有"计算过程"。

**用 CoT：**
```
Q: 小明有 5 本书，他买了 3 本，又送了 2 本给朋友，还剩几本？
A: 让我一步步算：
   初始：5 本
   买 3 本后：5 + 3 = 8 本
   送 2 本后：8 - 2 = 6 本
   答：还剩 6 本
```

CoT 强制模型把推理过程展开成 token 序列，每一步都基于前一步——这大大降低了错误率。

### CoT 为什么有效？

**根本原因**：Transformer 的 decoder 是**自回归**的——下一个 token 基于之前所有 token。CoT 把"隐式计算"变成"显式 token"，让模型有更多的计算步骤。

简单说：**CoT 给了模型更多的"思考时间"（以 token 形式）**。

### CoT 的三种模式

| 模式 | 做法 | 适用场景 |
|---|---|---|
| **Zero-shot CoT** | 加一句 "Let's think step by step" | 通用，什么都不用准备 |
| **Few-shot CoT** | 给 2-3 个带推理过程的例子 | 推理模式固定时效果最好 |
| **Auto-CoT** | 自动聚类问题 → 生成代表性问题 → 自动产生 CoT 示例 | 大批量同类问题时 |

**Zero-shot CoT 的黄金提示词**：
- `Let's think step by step.`（最泛用）
- `Before answering, let's break this down.`（适合复杂分析）
- `Think through this carefully, step by step.`（适合推理题）

### CoT 的边界

CoT 不是万能药。以下情况 CoT 反而有害：

1. **简单事实查询**：问"法国的首都是什么"不需要推理过程
2. **创意写作**：CoT 会让输出过于结构化、失去灵气
3. **模型本身太弱**：小模型（<7B）的 CoT 推理可能引入更多错误
4. **上下文已有足够信息**：不需要重复推理

## 3. 超越 CoT：更高级的推理策略

### ReAct (Reasoning + Acting)

CoT 只管推理，ReAct 把**推理 + 行动**交织在一起：

```
Thought: 我需要知道今天北京天气
Action: get_weather("北京")
Observation: 晴，25°C
Thought: 天气不错，可以推荐户外活动
Action: search("北京户外景点")
Observation: 颐和园、长城、奥森公园...
Thought: 我有了足够信息来回答用户了
Answer: 今天北京晴天 25°C，推荐去...
```

ReAct 是 **Function Calling 的灵魂**——模型在推理过程中决定何时调用工具、如何解读工具返回的结果。

### Tree-of-Thought (ToT)

CoT 是线性推理，ToT 是**树状搜索**：
- 每一步生成 N 个候选思路
- 评估每个思路的质量
- 只推进最好的 K 个分支
- 重复直到找到最优解

适用场景：需要探索多种可能性的问题——游戏策略、数学证明、架构设计。

**代价**：ToT 的 token 消耗是 CoT 的 10-100 倍，只适合高价值的推理任务。

### Self-Consistency

同一个问题跑 CoT 多次（temperature > 0），然后投票选最常见的答案。

**直觉**：如果一个问题有多种推理路径，但都指向同一个答案，那这个答案更可能是正确的。

## 4. 实践决策框架

拿到一个新 prompt 任务时，按以下顺序决定策略：

```
这个任务够简单吗？
├── 是 → Zero-shot，加一句 "Let's think step by step"
└── 否 → 输出格式有特殊要求吗？
    ├── 是 → Few-shot（2-3 个格式示例）
    └── 否 → 任务需要多步推理吗？
        ├── 是 → Few-shot CoT（2-3 个推理示例）
        └── 否 → 任务需要调用工具吗？
            ├── 是 → ReAct（推理+行动交织）
            └── 否 → ToT / Self-Consistency（高价值任务）
```

### 经验法则

1. **能简单不复杂**：Zero-shot CoT 是默认起点
2. **格式问题用例子解决**：不要用文字描述格式，直接给格式示例
3. **推理链示例质量 > 数量**：2-3 个好的 Few-shot CoT 示例比 10 个平庸的更有效
4. **CoT 不是免费的**：推理链消耗 token，简单任务不加 CoT
5. **temperature 0 用于测试和验证**：确保可复现

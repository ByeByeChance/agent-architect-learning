# LLM 的能力与边界

> 阅读：
> - GPT-4 Technical Report: https://arxiv.org/abs/2303.08774
> - Anthropic Model Card: https://docs.anthropic.com/en/docs/about-claude/models

## 1. LLM 擅长什么？

**文本理解与总结**
把长文档压缩成几个要点，准确率远超传统 NLP 方法。例如：把 20 页合同浓缩为一页摘要，抓住所有关键条款。

**翻译**
多语言互译，特别是常用语言组合（中-英、英-日等）。相比传统统计机器翻译，LLM 能更好处理语境和惯用语。

**代码生成**
从自然语言描述生成代码，是目前最成功的大模型应用场景之一。GitHub Copilot 使用 LLM 做代码补全，已被验证能显著提升开发者效率。例如：说"写一个排序函数"就能直接出代码。

**信息提取**
从非结构化文本中提取结构化数据，如人名、日期、金额、实体关系。例如：从一堆邮件中提取所有提及的订单号和金额。

**创意写作**
故事、文案、广告语、诗歌——虽然不总是最佳水平，但能快速产出可用的初稿。例如：给 5 个品牌 slogan 选项供选择。

**对话与客服**
理解多轮对话的上下文，给出合理回应。可以处理 FAQ、工单分类、情绪安抚等客服场景。

**知识问答**
对于训练数据覆盖领域的事实性知识，回答准确率高。例如：历史事件、科学原理、编程知识点。

## 2. LLM 不擅长什么？

**精确数学计算**
LLM 本质是模式匹配，不是计算器。大数乘法、多步运算容易出错。它预测"最可能的下一个数字"，而不是真正在计算。例如：问 12345 × 67890 可能得到错误答案。解法：用 Function Calling 调用真正的计算器。

**实时信息**
训练数据有截止日期，模型不知道训练后发生的事。问"今天天气"或"最新新闻"无法回答。解法：加 RAG 或搜索工具。

**精确记忆**
不能逐字回忆训练数据中的具体文档或引用。它学到的是"模式"而非"副本"。解法：把文档注入上下文。

**多步逻辑推理**
简单推理可以，但需要很多步（5+步）的复杂推理容易中途"跑偏"或遗漏条件。例如：复杂的逻辑谜题、多条件的法律分析。

**确定性输出**
同样的 prompt 可能每次结果略有不同（除非 temperature=0）。对需要绝对一致性的场景（如数据校验）不适合。

## 3. 幻觉 (Hallucination)

**什么是幻觉**
LLM 生成的内容看起来合理、流畅、有说服力，但事实上是编造的。例如编造不存在的论文标题和作者、虚构历史事件、给出看似精确但错误的数字。

**为什么会产生**
- LLM 的核心任务是"预测最可能的下一个 token"，不是判断"这是真是假"
- 训练目标是语言流畅性，不是事实准确性
- 当模型对某领域知识不足时，它会"脑补"一个语言上合理的答案，而不是说"我不知道"

**缓解手段**
- RAG（检索增强生成）：先检索真实文档，约束模型基于文档回答
- 要求模型引用来源/出处
- Temperature=0 降低随机性，减少"脑补"倾向
- Function Calling 让模型调用外部工具（搜索引擎、计算器）验证

## 4. Temperature 和 Top-P

| 参数 | 作用 | 低值效果 | 高值效果 |
|---|---|---|---|
| Temperature | 控制输出随机性，影响概率分布"平坦度" | 0 = 确定性输出，同样的输入永远一样的结果 | 1+ = 输出多样但可能不连贯，随机性强 |
| Top-P | 核采样，只从累积概率达到 P 的候选 token 中选取 | 0.1 = 只选最可能的几个词，输出保守 | 1 = 考虑所有可能，输出更丰富 |

- Temperature 影响 logits 分布的形状（拉平或锐化概率）
- Top-P 影响候选 token 的范围大小
- 创意写作用高 Temperature（0.7-1.0）、代码生成和事实问答用低 Temperature（0-0.3）

## 5. 决策框架：这个任务适不适合用 LLM？

**维度 1：对精确度的要求**
允许不完美的回答 → 适合用 LLM。必须 100% 准确（如财务报表）→ 需要人工校验或不用 LLM。

**维度 2：对实时性的要求**
不需要最新信息 → 适合。需要今天的数据 → 需要配合 RAG 或搜索工具。

**维度 3：任务的模糊度**
边界清晰、理解一致 → 适合。需要强逻辑推理或领域专长 → 需要拆解成子任务或不用 LLM。

**维度 4：成本接受度**
愿意为每次调用付费 → 适合。需要免费无限次调用 → 传统方法可能更合适。

**维度 5：是否有更简单的办法**
正则表达式就能解决 → 不用 LLM。需要理解语境和语义 → LLM 是正确工具。

## 补充知识点

**Scaling Law（缩放定律）**
"大力出奇迹"的数学依据。OpenAI 2020 年论文发现，模型性能与模型参数量（N）、训练数据量（D）、训练计算量（C）三个因素呈稳定的幂律关系。三者需要均衡增长，只增大模型而不增加数据会导致欠拟合。这解释了为什么 GPT-4 比 GPT-3 好——不是换了架构，而是"更大了"。

**涌现能力 (Emergent Abilities)**
当模型规模超过某个阈值后，突然出现训练目标中没有明确教过的能力——数学推理、代码理解、角色扮演等。小模型完全不会，大模型突然就会了。这说明扩大规模不仅提升已有能力，还会"解锁"新能力。目前这种现象的原因仍在研究中。

**In-Context Learning（上下文学习）**
LLM 不需要微调就能通过 prompt 中的几个例子快速学会新任务。这是 Transformer 架构在设计时完全没想到的"意外能力"——训练的目标只是预测下一个 token，但模型自己学到了"从例子中归纳规则"的能力。

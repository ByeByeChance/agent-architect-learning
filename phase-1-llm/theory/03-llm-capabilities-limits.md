# LLM 的能力与边界

## 1. LLM 擅长什么？

- **文本理解与总结**：把长文章压缩成几个要点，准确率远超传统 NLP
- **翻译**：多语言互译，特别是常用语言组合
- **代码生成**：从自然语言描述生成代码，GitHub Copilot 的生产力提升已被多篇论文验证
- **信息提取**：从非结构化文本中提取结构化数据（人名、日期、金额）
- **创意写作**：故事、文案、广告语——虽然不是最佳，但能快速提供初稿
- **对话与客服**：理解多轮对话上下文，给出合理回应
- **知识问答**：对于训练数据中涵盖的事实性知识，回答准确率高

## 2. LLM 不擅长什么？

- **精确数学计算**：LLM 本质是模式匹配，不是计算器。大数乘除、复杂公式容易出错
- **实时信息**：训练数据有截止日期，不知道"今天发生了什么"。需要 RAG 或搜索来弥补
- **精确记忆**：不能逐字回忆训练数据中的具体文档，这是建模方式决定的
- **多步逻辑推理**：简单推理可以，但需要很多步推理的复杂问题容易中途"跑偏"
- **确定性输出**：同样的 prompt 可能每次结果略有不同（除非 temperature=0）

## 3. 幻觉 (Hallucination)

**什么是幻觉**：LLM 生成的内容看起来合理，但事实上是编造的。比如编造不存在的论文、人名、数据。

**为什么会产生**：
- LLM 的任务是"预测最可能的下一个 token"，而不是"判断真假"
- 当模型对某个领域知识不足时，它会"脑补"平滑的答案而不是承认不知道

**缓解手段**：
- RAG（检索增强生成）：先检索真实文档，再基于文档回答
- 要求模型引用来源
- Temperature=0 降低随机性
- 用 Function Calling 让模型调用外部工具验证

## 4. Temperature 和 Top-P

| 参数 | 作用 | 低值 | 高值 |
|---|---|---|---|
| Temperature | 控制输出的随机性 | 0 = 确定性输出，每次都一样 | 1+ = 输出多样但可能不连贯 |
| Top-P | 核采样，只从累积概率达到 P 的 token 中选 | 0.1 = 只选最可能的几个词 | 1 = 考虑所有词 |

- Temperature 影响 logits 分布的形状（拉平或锐化概率分布）
- Top-P 影响候选 token 的范围
- 创意写作用高 Temperature，代码/事实问答用低 Temperature

## 5. 决策框架：这个任务适不适合用 LLM？

1. **对"精确"的要求**：允许不完美 → 适合。需要 100% 准确 → 不适合
2. **对"实时"的要求**：不需要最新信息 → 适合。需要今天的数据 → 需要 RAG
3. **任务的"模糊度"**：边界清晰、理解一致 → 适合。需要强逻辑推理 → 需要拆解
4. **成本接受度**：愿意为每次调用付费 → 适合。需要免费无限次 → 不适合
5. **是否有更简单的办法**：正则就能解决 → 不用 LLM。需要理解语境 → 适合

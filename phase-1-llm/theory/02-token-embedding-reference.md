# Token 与 Embedding — 标准参考

> 你的实验笔记保留在 `02-token-embedding.md` 中。

## 1. 什么是 Token？

**Token 是 LLM 处理文本的最小单位。** 它不是"字"也不是"词"——它介于两者之间。

### BPE (Byte Pair Encoding) 原理

BPE 是目前主流 LLM 使用的分词方法。它的训练过程：

1. 从字节级开始（每个字符是一个 token）
2. 统计相邻 token 对的出现频率
3. 把最高频的一对合并成一个新 token
4. 重复步骤 2-3，直到达到目标词汇量（如 GPT-4 约 10 万）

**结果**：高频词是完整 token（`the` → 1 token），低频词被拆成子词（`tokenization` → `token` + `ization` → 2 tokens）。这样平衡了词汇覆盖率和效率。

### 为什么需要 Tokenizer？

LLM 不能直接处理文字。它需要一个"翻译器"把人类语言转成它能理解的数字序列。Tokenizer 就是这个翻译器。

## 2. 为什么中文 token 数比英文多？

你的实验数据已经说明了：

| 输入 | Token 数 |
|---|---|
| "Hello, world!" | 5 |
| "你好，世界！" | 6 |

**表面原因**：中文没有空格分隔词，tokenizer 需要更多 token 来表达同样的意思。英文每个词 ≈ 1-1.5 tokens，中文每个字 ≈ 1-2 tokens。

**根本原因**：主流 tokenizer 的训练数据以英文为主。BPE 合并时，英文高频词（`the`, `is`, `hello`）有更高的合并优先级，更容易成为完整 token。中文的词频在训练语料中占比低，合并优先级低，所以被拆得更碎。

**影响**：同样的语义内容，中文消耗的 token 数大约是英文的 1.5-2 倍。这意味着：
- 中文 API 调用成本更高
- 中文的 context window 有效容量更小
- 同一个问题中英文各问一遍，中文消耗更多 token

## 3. Token → ID → Embedding 的流程

```
输入文字: "我学习 AI"
    ↓
Tokenizer: ["我", "学习", "AI"]     # 分成 3 个 token
    ↓
Token IDs:  [1234, 5678, 910]      # 查表转成数字 ID
    ↓
Embedding Matrix:                   # 每个 ID 对应一个向量
    ID=1234 → [0.12, -0.34, 0.56, ...]   (d_model 维向量)
    ID=5678 → [0.78, 0.11, -0.45, ...]
    ID=910  → [-0.23, 0.67, 0.34, ...]
    ↓
输入 Transformer 的是一组向量序列
```

**Embedding Matrix** 是一个形状为 `[vocab_size, d_model]` 的可学习矩阵，比如 GPT-3 的词汇量约 5 万，d_model = 12288，这个矩阵就是 50000 × 12288 的巨大参数表。

---

## 4. Embedding 做了什么？

Embedding 把一个离散的 token ID 映射到一个连续的、高维的向量空间。这个向量**编码了语义信息**。

**经典例子：king - man + woman ≈ queen**

在向量空间中做算术：
- `king(国王)` 的向量减去 `man(男人)` 的向量 = "去除男性属性"
- 再加上 `woman(女人)` 的向量 = "加上女性属性"
- 结果向量最接近 `queen(女王)` 的向量

这说明 Embedding 不仅记住了词的表面形式，还学到了**词与词之间的语义关系和属性**。相似的词在向量空间中距离更近。

## 5. Context Window（上下文窗口）

**Context Window 是模型一次能"看到"的最大 token 数量。**

比如 GPT-4 的 128K 上下文窗口意味着它一次最多处理 128,000 个 token（约 9 万英文词或 6 万中文字）。

### 为什么需要大上下文？

- 长文档分析（论文、合同、代码库）
- 多轮对话（保留完整对话历史）
- RAG（检索增强生成——需要把检索到的文档片段注入上下文）

### 为什么不是越大越好？

1. **注意力计算复杂度是 O(n²)**——上下文翻倍，计算量翻四倍。处理 128K token 比 4K token 慢约 1000 倍。
2. **"Lost in the Middle"效应**：研究发现，模型对上下文开头和结尾的信息利用最好，中间部分容易被忽略。窗口越大，中间越容易被忽视。
3. **成本**：输入 token 也计费。128K 的输入可能比 4K 贵几十倍。
4. **注意力稀释**：窗口越大，模型分配给每个 token 的"注意力"越少，可能遗漏关键信息。

> 所以不是越大越好。128K 是"有需要时可用"，但大多数任务 4K-8K 就够了。

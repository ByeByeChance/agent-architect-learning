# AI Agent 工程 — 可靠性建造总纲

**时间**：2026-06-17 ～ 2026-06-23  
**语言**：TypeScript/Node.js + React + 中文文档  
**最简总结**：六个阶段，回答一个问题。

---

## 核心命题

**LLM 是不可靠的。** 它会幻觉（编造不存在的事实）、它非确定（同样 prompt 两次回答不同）、它贵（一次 Agent 任务 5-20 次 API 调用）、它慢（单次调用数百毫秒，多步串行就是秒级）、它会忘（context window 中间的内容关注度最低）、它会被注入攻击（一段用户输入就能覆盖 system prompt）。

**那么，怎么在不可靠的基座上建造可靠的 Agent 系统？**

这不是一个能一次回答的问题。它是六层答案，每层解决不可靠的一个维度，六层叠在一起形成完整的可靠性栈：

```
层6：接受失败，建立安全网        ← 失败了可控
层5：用多点协作互相纠错           ← 一个错了另一个指出
层4：用事实锚定消除幻觉           ← 别自己编，看资料说
层3：用"程序外壳"约束模型         ← 给模型划定行为边界
层2：用确定性的工具替代推理        ← 工具是确定的，不做推理
层1：先承认不可靠                 ← 理解不可靠从哪来
```

**层与层之间的关系不是"并列"，而是"下层没有被上层替代"**——你不可能说"我有了 Phase 6 的熔断器所以 Phase 3 的 Prompt 校验就不需要了"。每层填补了特定类别的不可靠，不重叠、不可替代。

## 层 1：承认不可靠——到底什么不可靠？

**Phase 1: LLM 基础认知**

### 为什么这是第一层

大多数人对 LLM 的理解停留在一个黑箱："它能回答问题，有时候不对"。但如果你不知道为什么不对、什么情况下不对、对到什么程度，你就没有基础去解决后面的问题。

一个具体场景让你体会这个问题：

- 你用 DeepSeek API 生成了一段代码，结果看起来是对的
- 你用同样的 prompt 换了 OpenAI，结果不同——哪个对？
- 你调了 temperature 从 0 到 1——生成质量是更好了还是更不稳定了？
- 你让模型调用了 `get_weather` 工具——模型什么时候决定调用、什么时候决定不调用？

**如果你不能回答这些问题，你就没有资格说"我在用 LLM"，你在"被 LLM 用"。**

### 这一层给后面的每一层铺了什么路

Phase 1 做了三件事：

**第一，弄明白了模型到底怎么产出 token。** Transformer 的自回归性质意味着每个 token 的概率都建立在之前所有 token 的条件概率上。这不是"模型在想"，是"模型在采样"。Temperature=0 选概率最大的，Temperature=1 按分布随机选。这意味着**两个相同的 prompt 可以产生完全不同的输出**——这是非确定性的数学根源，不是 bug。

这个认知贯穿后面所有阶段：
- Phase 4 的 RAG 为什么需要检索而不是信任模型的内部知识？因为模型不是"知道"，是"生成"。
- Phase 5 为什么需要多个 Agent 交叉审查？因为单次 LLM 调用的输出是不可验证的——你拿不到"为什么这么答"。
- Phase 6 的对抗测试为什么需要 12 个用例反复跑？因为同样的注入攻击可能一次被拦一次不被拦——攻击者只需要一次成功。

**第二，弄明白了 Function Calling 的本质。** 模型不"理解"工具是什么。模型只是在输出的 token 序列中生成了一个特殊格式的 JSON：`{"name": "get_weather","arguments": {...}}`。这个 JSON 被你的代码拦截、执行、再把结果拼回 context。**模型全程不知道"真实的天气"——它只知道"一段文本说天气是这样"**。

这个认知直接决定了 Phase 2 的 MCP 设计——工具的定义（Tool Definition）是给模型看的"说明书"，工具的执行（Tool Execution）是你的代码，两者通过 JSON-RPC 解耦。模型从不执行工具，它只负责"建议调用什么工具"。

**第三，弄明白了 LLM 的四个核心局限：**

| 局限 | 表现 | 后果 | 由哪一层解决 |
|---|---|---|---|
| **幻觉** | 编造不存在的 API/数据/事实 | 输出不可信 | 层 4 (RAG 锚定) |
| **非确定性** | 相同 prompt 不同结果 | 无法回归测试 | 层 3 (Prompt 程序化) |
| **Context 限制** | 太多信息时中间内容被忽略 | 长对话质量下降 | 层 4 (记忆管理) |
| **脆弱性** | 一段注入文本就能覆盖指令 | 安全问题 | 层 6 (护栏) |

每个局限在后面都有对应的解决方案。但如果你在 Phase 1 没有深刻理解"为什么会有这个局限"，后面你会把解决方案用错场景。

### 关键产出物

- `theory/01-transformer-attention.md`：Attention 让信息传递从 O(n) 降到 O(1)，这是 Transformer 能 scale 的根基
- `theory/03-llm-capabilities-limits.md`：四个局限的详细分析
- `practice/agent/function-calling/`：模型不"知道"天气，它只知道"有一段文本说天气是这样"
- `practice/agent/experiments/`：Temperature 从 0 到 1 的输出变化——亲眼看到非确定性

## 层 2：用确定性的工具替代不确定的推理

**Phase 2: MCP 协议与工具构建**

### 这个阶段解决什么

Phase 1 的结论是：**模型不应该自己"想"事实，它应该调用工具获取事实。** 工具是确定性的——`get_weather("Beijing")` 返回的是真实天气，不是模型"觉得"的天气。

但如何让模型能调用任何工具而不需要每次都写新的集成代码？这就是 MCP（Model Context Protocol）要解决的问题。

### MCP 的本质：一个协议，不是一套工具

**MCP 不提供任何工具——它只定义"工具如何被发现"和"工具如何被调用"。**

```
┌──────────┐    JSON-RPC     ┌──────────┐
│  LLM     │   tools/list    │  MCP     │
│  (Client)│ ←─────────────→ │  Server  │ → 实际执行工具
└──────────┘   tools/call    └──────────┘
```

这个设计决策的意义：**Client 和 Server 互相不知道对方的存在**。Client 只知道"有一个叫 get_weather 的工具，参数是 city:string"，不知道它背后是 HTTP API 还是文件系统还是数据库。Server 只知道"我提供 get_weather 工具"，不知道哪个 LLM 在调用它。

这解决了工具集成的**N×M 问题**：传统方式下 N 个 LLM 要对接 M 个工具系统需要 N×M 个适配器。MCP 下只需要 N 个 Client + M 个 Server = N+M 个组件。

### 我们在 Phase 2 做的事情

写了 3 个 MCP Server——每个都是一个完整的、可独立运行的 TypeScript 进程：

1. **Hello Server**：最简实现——一个 `hello` tool + 一个 `echo` tool，证明 MCP 协议的最小可行体
2. **Weather Server**：Tool + Resource 协作——`get_weather` 是 tool（动作），`weather://{city}` 是 resource（数据），证明了 Tool 和 Resource 是两种不同的概念
3. **File Search Server**：环境变量权限控制——`SEARCH_ROOT` 决定搜索范围，证明了 MCP Server 可以有自己的安全策略

还写了一个 React 调试面板（3 个 Tab：Servers 列表 / Tool Call 记录 / Chat 交互），验证了"管理面、数据面、交互面"三个视角缺一不可。

### 这一层给层 3 铺了什么路

有了工具，**Prompt 就可以引用工具了**。System Prompt 可以说"当用户问天气时，调用 get_weather 工具"。这是从"回答问题"到"执行任务"的关键一步——模型不再是被动的回答者，而是主动的工具调用者。Phase 3 的 Prompt Engineering 就是在这个基础上让"工具调用决策"变得更可靠。

### 关键产出物

- `theory/01-mcp-protocol.md`：JSON-RPC、Tool/Resource/Prompt 三个概念的精确定义
- `servers/file-search-server/index.ts`：递归文件搜索 + 环境变量权限控制
- `dashboard/`：三 Tab 调试面板

## 层 3：用"程序外壳"约束模型的自由

**Phase 3: Prompt Engineering**

### 这个阶段解决什么

层 2 给了模型工具。但**工具只是能力，不是约束**。模型有了工具可以"正确地"做事，但——
- 它什么时候该用工具？什么时候不该？
- 它的输出格式是什么？调用方怎么解析？
- 它的行为边界在哪？什么绝对不能做？

这些问题的答案是 **Prompt**。Prompt 是模型的"程序外壳"——它约束模型的行为，规定输入输出格式，定义什么能做、什么不能做。

### Prompt 是程序，不是对话

Phase 3 最核心的观点转变：**Prompt Engineering 不是聊天技巧，是软件工程。**

把这个观点想清楚：
- 一段 Python 程序有输入、输出、逻辑、异常处理、边界条件
- 一段 System Prompt 有输入（user message）、输出（completion + tool calls）、逻辑（规则、约束、步骤）、异常处理（"如果你不确定，回答不知道"）、边界条件（"不要讨论政治话题"）

两者的区别只在于：Python 程序的运行时是 CPython（确定性），Prompt 程序的运行时是 LLM（非确定性）。**但工程化的做法是一样的——版本管理、测试、CI/CD——就因为运行时非确定，才更需要工程化。**

CoT（Chain-of-Thought）的机制进一步强化了这个观点。"让我们一步一步思考"不只是 trick——**每个 CoT token 都让模型多做一次前向传播**。Transformer 的自回归性质意味着：token₂ 的计算依赖 token₁ 的 hidden state，token₁₀ 的计算经过了 9 轮 self-attention。所以 CoT **等价于给了模型更多的计算步骤**。这不是心理暗示，是计算架构决定的。

### 我们在 Phase 3 做的事情

三个工程化实践：

**1. Prompt 版本化管理（prompt-vault）**
- 每个 System Prompt 是一个版本化文件，有 `version` 字段
- CLI：`list`（列出）/ `show <id>`（查看）/ `diff <id1> <id2>`（对比）/ `validate <id> --smoke`（校验 + 冒烟测试）
- 用 dummy 值渲染模板检测括号不匹配（学习 LangChain 的 `checkValidTemplate`）

**2. 回归测试套件（test-suite，20 个用例，5 类）**
- 格式测试：输出是否符合预期 JSON schema
- 安全测试：是否拒绝 prompt 注入请求
- 行为测试：是否在不确定时回答"我不知道"
- 对抗测试：是否被 jailbreak prompt 绕过
- 边界测试：极端输入（空输入、超长输入）是否崩溃

**3. 多模型对比（multi-model-bench）**
- 10 个相同的 Prompt × 3 个模型（DeepSeek/OpenAI/Anthropic）
- 统一评估框架：规则断言 + LLM-as-Judge（llmRubric）
- 区分 graderError（评估系统出错）和内容失败（模型输出不达标）

### 这一层给层 4 铺了什么路

有了结构化的 System Prompt，才能定义 Agent 的"角色"。Phase 5 的 Orchestrator、Code Generator、Code Reviewer 之所以是不同的 Agent，就是因为它们的 System Prompt 定义了完全不同的角色和行为边界。**如果 Prompt 不是工程化的，多 Agent 就是"多段不同的自然语言混在一起"——无法管理、无法测试、无法优化。**

### 关键产出物

- `theory/01-few-shot-cot.md`：CoT 为什么有效——每个 token 一次前向传播 = 更多计算
- `theory/02-system-prompt-design.md`：System Prompt 四大职责像 CSS——全局到局部层层递进
- `practice/prompt-vault/`：Prompt 版本仓库 + CLI
- `practice/test-suite/`：20 用例，5 类，引擎含 graderError/score/llmRubric
- `practice/multi-model-bench/`：10 prompt × 3 模型统一评估

## 层 4：用事实锚定消除幻觉

**Phase 4: Agent 记忆与 RAG**

### 这个阶段解决什么

层 3 让模型"按规则做"，但**规则再清晰，模型也不知道事实**。System Prompt 说"用最新文档回答"，但最新文档不在模型的参数里——它训练于过去的某个时间点，之后的事它不知道。

**RAG 的核心思想：不给模型靠内部记忆回答的机会。给它事实，让它"照着说"。**

这里有一个关键区分——**Embedding 模型和 LLM 是两种完全不同的模型**：

| | Embedding 模型 | LLM |
|---|---|---|
| **架构** | Encoder-only (BERT 系) | Decoder-only (GPT 系) |
| **做什么** | 把文本变成向量 | 把向量（+prompt）变成文本 |
| **大小** | 几百 M 参数 | 几百 B 参数 |
| **本质** | "理解"语义 → 向量 | "生成"文本 → 返回 |
| **在 RAG 中的角色** | **找到**相关文档 | **基于**文档回答 |

很多人把 RAG 当成"一个东西"——对他们来说 RAG = "查资料然后回答"。但真正理解 RAG 的人知道，**RAG 是两个独立系统的协作**：Embedding 模型负责检索（确定性高、速度快、便宜），LLM 负责生成（不确定性高、速度慢、贵）。把检索和生成解耦，是 RAG 设计的核心。

### Chunking——RAG 最被低估的工程问题

RAG 的质量天花板不在 LLM，在 Chunking。Chunk 太大 → 检索不准（"这个 chunk 包含 8 个话题，相关性分很低"）。Chunk 太小 → 缺乏上下文（"这段代码没有函数名，不知道是干什么的"）。

我们的 `RecursiveChunker` 从段落 → 句子 → 词 → 字符逐层降级，保证每个 chunk 在大小限制内尽可能完整。`overlap` 参数不是可选的——不加 overlap，chunk 边界恰好在关键信息处断开，LLM 收到不完整的上下文就会答非所问。

### 三层记忆——解决"Lost in the Middle"

LLM 的 context window 有一个反直觉的属性：**它对 prompt 中间位置的信息关注度最低，对开头和结尾最高。** 这叫"Lost in the Middle"。所以不是 context window 越大越好——你可以塞 200K tokens，但中间的 100K tokens 模型几乎"看不见"。

我们的三层记忆管理器直接回应这个问题：
- **工作记忆**：当前对话的最近 N 轮，放在 context 末尾（注意力最高）
- **短期记忆**：超过 N 轮的内容压缩为摘要，放在 context 开头（注意力次高）
- **长期记忆**：跨会话的知识存入向量库，只在需要时检索注入

**assembleContext() 是记忆管理器最核心的方法**——它决定什么信息以什么优先级进入 LLM 的有限 attention 预算。这个决策直接影响 Agent 的"认知能力"。

### 这一层给层 5 铺了什么路

多 Agent 系统中，Agent A 的发现如何被 Agent B 检索？如果每个 Agent 的记忆是独立的，那多 Agent 就失去了协作的意义。Phase 4 的向量库和记忆管理器是 Agent 间共享记忆的基础——共享的向量库让 Agent B 能检索到 Agent A 产生的知识，而不需要 Agent A 和 Agent B 直接通信。

### 关键产出物

- `theory/01-embedding-vector-search.md`：Embedding 模型 ≠ LLM，Encoder-only vs Decoder-only
- `theory/02-rag-architecture.md`：5 种 Chunking 策略 + Small-to-Big 检索
- `practice/rag-pipeline/`：Chunker → Embedder → Retriever → Generator 完整管线，来源标注验证
- `practice/memory-manager/`：三层记忆 + assembleContext，工作/短期/长期

## 层 5：用多点协作互相纠错

**Phase 5: 多 Agent 架构**

### 这个阶段解决什么

前四层让单个 Agent 尽可能可靠——它有工具、有 Prompt、有记忆。但**单个 LLM 调用的天花板是：你永远不知道它这次的回答是否可靠**。同样的 Prompt + 同样的上下文 + 同样的工具，两次调用可能给出品质完全不同的结果。

**多 Agent 的答案：不谈提高单个模型的可靠性，而是让多个模型互相检查。**

这个思想来自软件工程的一个古老智慧：代码审查。即使最好的程序员也无法保证自己写的代码 100% 无 bug——不是因为不努力，而是因为"审查自己的代码"存在认知盲区。所以用第二双眼睛。

Agent 也一样。一个 Code Generator 生成了代码，自己审查自己的代码基本看不出问题——因为它的输出就是它的"认知范围"。但 Code Reviewer 用完全独立的 System Prompt 来看这段代码——它是另一个"角色"，有不同的关注点。

### 多 Agent 不是"多调几次 LLM"

这是一个关键区分。一个 Agent 重试 3 次，每次用的是同一个 System Prompt、同一个角色，3 次之间的差异只是 temperature 导致的随机性——这像是"问了同一个人 3 遍"，如果这个人搞错了，问 3 遍还是搞错。

多 Agent 是给每次调用一个**不同的、狭窄的角色**：

```
Orchestrator：  拆解任务 → 分配 → 验证结果格式 → 汇总
    ↙                                ↖
Code Generator：生成代码     Code Reviewer：审查代码
    ↓                                    ↑
Test Writer：  生成测试 ──────→ 测试反馈代码
```

每个 Agent 的 System Prompt 只定义了非常狭窄的职责——Code Generator 不知道什么是"审查"，它只知道"根据规格生成代码"。角色越窄，幻觉空间越小。

### Orchestrator 预分解 vs AutoGen 动态选择——两种哲学

我们在实践中选择了"让 LLM 提前分解任务"（确定性路线），而不是"让 LLM 动态选择下一个发言者"（AutoGen 路线）。两者各有所长：

- **预分解**：Orchestrator 一次性把所有任务分好 → 拓扑排序找到依赖 → 按序执行。适合工程任务（"写一个工具函数、审查、测试"），因为任务结构是固定的。
- **动态选择**：每步完成后 LLM 决定"下一步谁发言"。适合探索性任务（"我不知道哪个 Agent 最适合回答这个问题"），因为任务结构不固定。

**不迷信框架，根据场景选。** 这是 Phase 5 最重要的工程判断。

### HITL——当 Agent 互相检查也不够时

多 Agent 协作降低了一定比例的错误。但有些决策（删除文件、DROP TABLE、执行远程命令）需要人的判断。HITL Gate 的核心设计是**粒度的博弈**：

- 太细（每次 API 调用都审批）→ 用户会关掉它 → 等于没有
- 太粗（只在重大决策审批）→ 中间风险的盲区 → 等于漏了
- **三级粒度**（CRITICAL → 必审 / WARNING → 建议审 / INFO → 自动过）是最小可行粒度

### 这一层给层 6 铺了什么路

多 Agent 系统让"可靠性"从单点提升到多点。但它引入了一个新问题：**多 Agent 协调本身成了新的单点故障。** Orchestrator 宕了全链都停、Agent 间的消息丢了导致任务卡死。层 6 的熔断器、降级链、优雅关闭就是为这些问题设计的。

- `theory/01-multi-agent-topology.md`：四种拓扑（编排/对等/层级/混合）
- `practice/orchestrator/`：分解→拓扑排序→执行→验证→汇总，3 个 Worker 任务 3/3 通过

## 层 6：接受失败不可避免，建立安全网

**Phase 6: Agent 产品化与治理**

### 这个阶段解决什么

前五层极大地提高了 Agent 的可靠性。但有一个不可逃避的事实：**没有任何一层能 100% 消除不可靠。** LLM API 会超时（435）、工具会返回错误、编排器会中途崩溃、注入攻击会绕过护栏。产品化的核心不在于"让失败不发生"——在于**"失败发生时，损害最小"**。

这是和很多人的直觉相悖的：**可靠性不是不失败，是失败了可控。**

### 成本治理——失败有成本，成本必须可见

LLM 的每次失败重试都会产生扣费。一个陷入重试循环的 Agent 可能在 5 分钟内烧掉 $5——不是模型故意浪费钱，而是**没人告诉它它已经超预算了**。

我们的成本追踪体系从三个层面解决这个问题：

1. **Token → Dollar 翻译**（PricingEngine）：token 是工程师的语言，美元是公司的语言。`35,000 tokens × $0.003/1K = $0.105` 比 `用了 35000 tokens` 有意义一百倍。

2. **五级预算树**（BudgetManager）：组织 → 团队 → 用户 → Key → Tag。子节点花费向上传播——Alice 花超了，不仅 Alice 被限制，她的团队也受影响。这不是"惩罚"，是"让成本被责任人感知"。

3. **Cost-per-Outcome**（CostOutcomeTracker）：不问"花了多少钱"，问"每完成一件事花了多少钱"。"生成代码审查 $0.42/次"和"修复审查发现的问题 $0.18/次"——这个差异告诉你哪些环节该自动化、哪些该人工。

### 信任与安全——失败有受害者，必须保护

Agent 安全的核心难题是：**攻击者只需要一次成功。** 而 LLM 的非确定性意味着同样的注入攻击可能第一次被拦截，第二次就绕过。所以安全护栏必须做纵深防御——单层防护是不足的。

**我们的护栏管线 12/12 对抗测试通过，但核心设计决策是：用 regex 不用 LLM。** 原因不是 LLM 检测不了注入，而是：

- 生产护栏必须 <50ms → LLM 调用 200-500ms
- 生产护栏必须 $0 → LLM 调用每次 $0.01
- 最重要的——**用 LLM 检测 LLM 是循环信任问题**：你怎么保证检测 LLM 本身不会被注入？

regex 虽然覆盖不到语义变体（把 "ignore previous instructions" 改成 "let's start fresh"），但它覆盖了 80% 的常见攻击且零延迟、零成本、100% 确定性。**安全不需要完美——攻击方也不是完美的。**

### 生产韧性——失败传播时，必须有墙

一个 LLM API 超时如果不加控制，会连锁反应：超时 → 线程阻塞 → 请求堆积 → 内存耗尽 → 整个服务不可用。**熔断器做的事很简单：连续失败 N 次后，"不再尝试，立即返回错误"。** 这比"等 30 秒然后超时"要好——因为快速失败不消耗资源。

CLOSED → OPEN → HALF_OPEN 三态转换的精妙之处在 HALF_OPEN：熔断不应该"永远断开"。冷却期过后允许少量探测请求——如果探测成功就全部恢复，如果探测失败就继续熔断。**不给恢复机会的熔断等于手动关停服务。**

### 审计日志——失败要追查，必须不可篡改

Agent 的自主性意味着它可能在无人注意时做了危险操作。审计日志是事后追查的唯一手段。**如果日志可以被篡改，追查就失去了意义。**

我们用 SHA-256 哈希链实现防篡改：每条日志的 `previousEntryHash` = 上一条日志 `output_hash` 的 SHA-256。改任何一条，整个链都断。这不需要区块链、不需要外部服务、不需要数据库——只用了 Node.js 的 `crypto` 模块。**哈希链是"不可抵赖"的最轻量实现。**

### 金丝雀发布——新模型不可靠，必须在有限流量上证明

模型升级（Claude 3.5 → Claude 4、GPT-4o 新版）带来的质量变化不是"变好了"或"变坏了"这么简单。新模型可能在某些场景下更好、某些场景下更差。**金丝雀发布把"模型升级"从一次赌博变成一次渐进验证**——5% → 25% → 50% → 100%，每一步对比行为质量，差就回滚。

### 这一层和其他五层的关系

层 6 不是"加功能"。**如果前五层是你造的汽车（引擎、轮子、方向盘、刹车、导航），层 6 是保险带、安全气囊、行车记录仪、仪表盘。** 你完全可以不要这些开着车上路——只是出了事后果会更严重。

## 完整的可靠性栈：六层如何协作

没有任何一层能独立工作。真实场景中，一个用户请求穿过六层的完整路径是：

```
用户："帮我写一个用户认证模块"

层6 输入护栏 ←──── 检查注入 + PII + 毒性
    ↓
层3 System Prompt ←─ 定义角色、输出格式、工具使用规则
    ↓
层5 Orchestrator ←─ 分解为 3 个子任务（代码生成 / 审查 / 测试）
    ↓
层2 MCP Server ────→ 工具调用（读取项目文件、写入代码）
    ↓
层4 RAG ───────────→ 从向量库检索项目现有代码风格、最佳实践文档
    ↓
层1 LLM API ──────→ 生成代码（可能有幻觉、可能超时）
    ↓
层5 交叉审查 ──────→ Code Reviewer Agent 审查生成的代码
    ↓
层6 输出护栏 ←───── 检查代码安全性（DROP TABLE? rm -rf?）
    ↓
层5 HITL Gate ────→ 如果是生产数据库操作 → 请求人工审批
    ↓
层6 成本追踪 ─────→ 记录 token 使用、计算 cost-per-outcome
    ↓
层6 审计日志 ─────→ SHA-256 哈希链写入审计轨迹
    ↓
用户收到审查结果
```

如果一个环节失败（比如 LLM API 超时）：
```
熔断器打开发 → 降级到备用模型 → 重试 → 成功 → 熔断器半开 → 探测后恢复
                              ↓
                         如果也失败 → 优雅降级（告知用户失败 + 保留已完成子任务结果）
```

**这就是可靠性栈：六层、每层不同维度、协作但不重叠。**

## 九个关键洞察（贯穿六层）

1. **模型不是在"想"，是在"采样"**（层 1）——每个 token 是概率分布上的一次采样。Temperature=0 选概率最大的，Temperature=1 按分布随机选。这意味着非确定性是数学性质的，不是 bug。

2. **工具是确定性的，推理是不确定的**（层 1→2）——MCP 让模型不"推理"事实，而是调用工具获取事实。工具的返回值是确定的（天气 API 返回 25°C 就是 25°C）。

3. **Prompt 是程序，不是对话**（层 3）——System Prompt = 类型定义 + 约束规则 + 异常处理。运行在非确定性的 LLM 上，所以比普通程序更需要测试。

4. **CoT 不是 trick，是给了模型更多计算步骤**（层 3）——自回归意味着 token₁₀ 经过 9 轮 self-attention。这是一个机制问题，不只是一个 prompt 技巧。

5. **Embedding 模型和 LLM 是两种模型，RAG 是两个系统的协作**（层 4）——Encoder-only 做检索（快、便宜、确定），Decoder-only 做生成（慢、贵、不确定）。误解这个等同的是绝大多数 RAG 失败的原因。

6. **多 Agent 不是同一问法问多遍，是给窄角色**（层 5）——每增加一个 Agent 不是为了"多一次机会"，而是为了"多一个视角"。角色越窄幻觉空间越小。

7. **成本治理本质是把成本分配给价值**（层 6）——cost-per-token 告诉你"花了多少"，cost-per-outcome 告诉你"值不值"。

8. **安全护栏不需要 LLM——用 LLM 检测 LLM 是循环信任**（层 6）——regex 覆盖 80% 攻击、<1ms、$0、100% 确定。语义覆盖不足的地方用第二层 LLM-based 防线补，不是用 LLM 替代 regex。

9. **可靠性不是不失败，是失败了可控**（层 6）——熔断器接受 API 会挂、降级链接受模型会超时、HITL 接受 Agent 会犯错、审计日志接受事后要追查。
